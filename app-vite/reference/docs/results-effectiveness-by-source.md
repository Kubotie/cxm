# Source Context別 Results効果指標・表示優先度・次アクション仕様

## UI設計思想

> **Resultsは単なる送信履歴画面ではなく、送信後に何が起きたかを見て、次に何をすべきか判断する画面**

- **Source Contextに応じて、何を「効いた」とみなすかを変える**
- **配信結果 → 接点反応 → 次アクション示唆の3段階で効果を見せる**
- **結果をCompany / Project / User / Workflow / Libraryに還元できるようにする**

---

## 効果の3段階定義

すべてのsource contextで、以下の3段階で効果を評価します:

### 1️⃣ **配信結果（Delivery）**
- ✅ Sent（送信成功）
- ❌ Failed（送信失敗）
- ⚠️ Unresolved（送信不可）
- 🔶 Partial Success（一部成功）

### 2️⃣ **接点反応（Engagement）**
- 💬 Reply（返信）
- 👁️ Open（開封）
- 🖱️ Click（クリック）
- ⭐ Reaction（リアクション）
- 📊 Status Change（状態変化）

### 3️⃣ **次アクション示唆（Next Action）**
- 🔄 Follow-up必要性
- 🎯 再設計必要性
- ✏️ Content / Template改善必要性
- 🔍 Company / Project / User深掘り必要性

---

## 共通必須表示項目

すべてのsource contextで最低限表示する項目:

### **配信サマリー**
- ✅ Source Context
- ✅ Channel（Email / Slack / Intercom / Chatwork）
- ✅ Audience Scope（Company / Project / User）
- ✅ Sent Time（送信日時）
- ✅ Target Count（対象数）
- ✅ Resolved Recipients Count（送信可能数）
- ✅ Sent Count（送信成功数）
- ✅ Failed Count（送信失敗数）
- ✅ Unresolved Recipients（送信不可対象）

### **反応サマリー**
- ✅ Open Rate（開封率）
- ✅ Click Rate（クリック率）
- ✅ Reply Count（返信数）
- ✅ Recent Reactions（最近の反応）

### **関連エンティティ**
- ✅ Linked Company / Project / User
- ✅ Linked Action
- ✅ Linked Content
- ✅ Linked Evidence

### **次アクション**
- ✅ Next Action Hints（次アクション候補）

## 1. From Inbox

### 📋 Source Context
問い合わせ・アラート・チケットへの返信結果

---

### 効果として見たいもの

#### **✅ 最重要効果指標**
1. **返信できたか**
   - 配信成功 / 失敗
   - チケット返信完了

2. **相手から反応があったか**
   - 顧客からの返信（Reply）
   - 再問い合わせ
   - 満足度反応

3. **問い合わせ・アラートの状態が前進したか**
   - Open → Resolved
   - Open → In Progress
   - Resolved → Reopened（悪化）

4. **再対応が必要か**
   - Follow-up必要性
   - エスカレーション必要性
   - 技術調査継続必要性

#### **⚠️ 注意すべき失敗パターン**
- Failed / Unresolved
- 返信後の再問い合わせ（解決していない）
- チケット再オープン

---

### 優先表示する結果

#### **クイックスタット（上部・最優先）**
```
┌─────────────────────────────────────────────┐
│ 📧 From Inbox: 問い合わせ返信結果           │
│ ─────────────────────────────────────────── │
│ ✅ 送信成功: 1件                            │
│ 💬 顧客返信: あり (2時間後)                 │
│ 📊 チケット状態: Open → Resolved           │
└─────────────────────────────────────────────┘
```

#### **優先表示項目（上から順に）**
1. **送信結果**
   - ✅ 送信成功 / ❌ 送信失敗
   - 送信日時

2. **顧客返信有無**
   - 💬 Reply有無
   - 返信内容プレビュー
   - 返信までの時間

3. **チケット / Inquiry状態変化**
   - Open → Resolved
   - Open → In Progress
   - Resolved → Reopened
   - 状態変化タイムスタンプ

4. **Reaction Time**
   - 顧客が最初にReplyするまでの時間
   - 問い合わせ受信から解決までの合計時間

5. **Follow-up必要性**
   - ⚠️ 再問い合わせあり
   - ⚠️ 未解決のまま
   - ✅ 解決完了

6. **再オープン可能性**
   - リスク評価（High / Medium / Low）
   - 予防的Follow-up推奨

---

### 次アクション

#### **推奨CTA（優先順位順）**
1. 🔙 **「Inboxに戻る」**
   - Inbox一覧へ
   - 他の未対応問い合わせへ

2. 📝 **「Follow-up Actionを作成」**
   - さらなる技術調査
   - エスカレーション
   - 追加サポート

3. 👤 **「Userを見る」**
   - 問い合わせユーザーの詳細
   - 過去の問い合わせ履歴

4. 🏢 **「Company / Projectで再確認」**
   - 関連するCompany / Project文脈の確認
   - 他の関連問い合わせ

5. 📄 **「Help / FAQ更新候補としてContentに送る」**
   - よくある質問として記録
   - Contentライブラリに追加

6. 📊 **「元の問い合わせを見る」**
   - Inbox内の該当itemへ戻る

#### **状況別の次アクション推奨**
```
┌─────────────────────────────────────────────┐
│ 🤖 AI推奨: 次のアクション                   │
│ ─────────────────────────────────────────── │
│ ✅ チケット解決完了                         │
│ → クローズして完了                          │
│                                             │
│ ⚠️ 顧客から再返信あり                      │
│ → Follow-up Actionを作成                   │
│ → 技術調査を継続                            │
│                                             │
│ ⚠️ 2日経過しても返信なし                   │
│ → Follow-upメール送信を検討                 │
└─────────────────────────────────────────────┘
```

---

### 避けるべき見せ方

#### ❌ **NG表示パターン**
1. **マーケティング施策の指標を主役にしすぎる**
   - Open Rate / Click Rateを最優先表示しない
   - 問い合わせ返信なのに「キャンペーン効果」的に見せない

2. **Reply起点なのにクリック率中心に見せる**
   - クリック数より、顧客返信や問い合わせ解消を優先

3. **一括施策のような見せ方**
   - セグメント分析を前面に出さない
   - 個別対応の結果として見せる

4. **送信成功だけで完了とみなす**
   - 送信成功 ≠ 問い合わせ解決
   - チケット状態変化まで追跡

#### ✅ **理想的な見せ方**
- 問い合わせ解消が最優先指標
- 顧客返信が効果の中心
- チケット状態変化が明確
- 再対応必要性が分かる
