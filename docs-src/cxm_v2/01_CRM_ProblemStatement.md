# CXM Problem Statement


## 1. 背景と前提
本プロジェクトは、Ptengine事業における顧客体験を、担当者の勘や分断されたツールに依存せず、組織として再現可能に運用するためのCXM（Customer Experience Management）基盤を構築する取り組みである。

Ptengineの顧客接点は、Slack/Chatwork/Intercom/メール/議事録/チケット（CSE）/製品利用ログ/CRM/Web接触などに分散しており、顧客の「いまの状態（迷い・期待・納得・不満・信頼・拡張意向）」は断片的にしか観測できない。この結果、顧客の前進（導入・オンボーディング・活用・拡張）に対する介入が遅れたり、過剰になったり、的外れになることで、体験の損失（Loss of Experience）が発生している。

CXMは、これらの分散ログを統合し、Signal（Health/Risk/Opportunity）として構造化し、顧客フェーズ（Phase 1-10）と接続して、顧客の前進に必要な介入（Action）を組織的に実行・学習するための「顧客前進OS」である。

本CXMのv1は、組織の主リソースであるCSMが担当顧客に対して価値を出せること、さらにCSMが対応しない顧客群に対しても機会を落とさないことを中心に据える。Supportは摩擦除去により顧客前進を止めない補完レイヤーとして機能し、CRMはCSMが見切れない顧客・局面を動的セグメンテーションによって支援しつつ、規模的顧客育成も担う。GTMは製品価値の翻訳・Enablementを担い、顧客理解の学びをもとに再現可能な価値提供の型を整備する。

また本v1ではAIを「意思決定の代行」ではなく、顧客理解（Log/Signal/Phase）に基づくコンテンツ生成・編集・導線設計（記事/資料/手順/ヘルプ/テンプレ）を加速するために活用する。Phase（1-10）はSSOTを前提とし、AI/システムは推定・提案を行うが自動上書きはしない。

---


## 2. 現状の課題（Problem）
現状の課題は大きく「顧客理解の断片化」「介入の不整合」「学習の欠如」に分解できる。

### 2.1 顧客理解が断片化しており、前進のタイミングを逃す
顧客の意思決定や前進は、担当者の会話・社内の稟議・利用者の詰まり・運用の習慣化などの連続で起きる。しかし現状では、その兆候がログに散在しており、担当者は必要な文脈を探すこと自体に時間を取られる。結果として、顧客の「いまの意図」や「未解決の論点」を組織として共有できず、介入が遅延する。

### 2.2 Signal（Health/Risk/Opportunity）が介入に接続されず、検知がアラートで終わる
利用低下や未解決チケット、拡張兆候などの重要な変化を検知しても、「誰が（どのレイヤーで）」「何を狙って」「どの証拠を根拠に」介入するかが標準化されていない。結果として、アラートが単なる通知に留まり、顧客の変化につながる実行（Action）へ結びつかない。

### 2.3 CSMが価値を作るための“組織攻略”が可視化されず、属人化する
重要顧客では、担当者だけでなく上長・経営・他部門を巻き込む合意形成が成果を左右する。しかし現状では、誰がキーマンで、誰が推進者/阻害者で、どの接点が不足しているかが構造化されず、引き継ぎやチーム攻略が難しい。結果として、CSMの成功が属人的になり、再現性とスケールが損なわれる。

### 2.4 Support/CRMの補完介入が“顧客理解”と接続されず、体験が分断される
Supportは瞬間的な解決で顧客の前進を止めない役割を担うが、問い合わせの背景文脈（Phase/Signal/利用状況）が不足すると、速くても刺さらない解決になり得る。CRMはCSMが見切れない顧客をスケールで支援するが、顧客の動的状態に同期したセグメンテーションとコンテンツが不足すると、配信が過剰・遅延・的外れになり、信頼が積み上がらない。

### 2.5 コンテンツ作成が人力に依存し、タイムリーな介入ができない
顧客の状態に合わせた記事・資料・手順・ヘルプ導線・テンプレは本来介入の中核だが、作成が人力で分断され、必要なときに必要な形で用意できない。結果として介入の速度と品質が頭打ちになり、顧客の前進の波に追随できない。

### 2.6 介入結果が学習として蓄積されず、改善が循環しない
「どのSignalに対して」「どのコンテンツ/介入が」「どの顧客に効いたか」が記録されないため、成功・失敗が個人の経験に留まる。結果としてPlaybook化が進まず、施策改善も属人的になる。

## 3. 目指す状態（Outcome / Success）
本CXMが目指すのは、顧客の前進を「観測→理解→判断→介入→学習」のループで増幅し、組織として再現可能にする状態である。特にv1では、CSMが担当顧客で価値を出せることを起点に、Support/CRMが補完介入として連動し、成果が蓄積されていく状態を作る。

### 3.1 顧客理解が一箇所に集まり、意思決定が速く正確になる
- Company/Project/UserのScopeを明確にした上で、ログ（証拠）、Signal、Phaseが統合される
- 3秒で「今この顧客をどう扱うべきか（最重要論点/リスク/機会/次アクション）」を把握できる
- PhaseはSSOTを前提に、推定と確定が区別され、判断ログとして残る

### 3.2 Signalが“介入”に自動的につながり、実行が標準化される
- Health/Risk/Opportunityの変化が検知されたとき、推奨介入レイヤー（CSM/Support/CRM/GTM）が明確になる
- 介入（Action/Sent）はOwner/Target/Evidenceを持ち、実行と責任が追える
- “通知で終わらず”、次の行動に接続される

### 3.3 AIがコンテンツ供給を加速し、介入の速度と品質が上がる
- 顧客理解（Log/Signal/Phase）に基づき、記事/資料/手順/ヘルプ導線/テンプレをAIがドラフト生成できる
- 人が承認・編集して使うことで、ブランドと品質を担保しながらスピードを上げる
- CSMは成果共有・Expand提案の準備が速くなり、Supportは即解決とナレッジ化が進み、CRMは動的コンテンツ配信が可能になる

### 3.4 学習が蓄積され、再現可能なPlaybookが増える
- 「どのSignalに対して、どの介入/コンテンツが効いたか」がImpactとして記録される
- 成功パターンがPlaybook化され、次の顧客に再利用できる
- 属人性が減り、組織として顧客前進を加速できる

### 3.5 v1の成功条件（定性的）
- CSMが日常的にCompany Detailを使い、理解→判断→介入（Sent/Todo）を迷わず回せる
- Supportが文脈を参照して即解決でき、ナレッジ化が進み始める
- CRMが少数の動的セグメントで“刺さるコンテンツ介入”を実行し、学習できる

## 4. 対象ユーザーとJTBD（役割の中心性）

本CXMのv1は、組織の主リソースであるCSMが「担当顧客に対して価値を見出せる」ことを中心に据える。
CRMは、CSMを補完し、CSMが見切れない顧客や局面を動的セグメンテーションで支援する役割を担う。加えて、規模的顧客の育成（Engagementまでの前進支援）も兼ねる。
Supportは顧客の摩擦除去を担い、CSMが深く関与できない局面でも顧客前進を止めないための重要な補完レイヤーとなる。
GTMは製品価値の翻訳・Enablementを担うが、v1においては中核価値（CSMが顧客価値を作る）を成立させた後に拡張する。

### 4.1 Primary User（v1）
- CSM（担当者）/ CSM Manager（司令塔・支援設計）
  - 顧客の状態理解（Phase/Signal/Timeline）を3秒で行い、今週の最重要論点を確定し、次の介入（Action）を意思決定・実行できること。
  - 組織攻略（Stakeholder/関係性/合意形成）を進め、価値の定着とExpandへつなげられること
  - 毎日の顧客の変化をシグナルを通してリアルタイム把握し、即対応に繋げられること。

### 4.2 Secondary Users（v1）
- Support / CSE（摩擦除去・顧客中心体験の担保）
  - 顧客の「困った瞬間」「詰まり」を最短で解消し、顧客の前進速度を落とさないこと。
  - 問い合わせ・チケット・障害対応を、顧客状態（Signal / Phase / 利用状況）の文脈に紐づけて理解し、適切な対応に接続できること。
  - 解決を単発で終わらせず、ナレッジ化・再発防止・体験改善へ還元し、Ptmindの顧客中心文化を接点で体現すること。
  - 必要に応じてCSMへエスカレーションしやすく、引き継ぎロス（文脈欠落）を減らせること。

- CRM / Lifecycle Ops（補完介入・規模的育成）
  - CSMが直接深く関与できない顧客や局面に対して、動的セグメンテーション（Signal / Usage / 行動）に基づく有益な情報・提案を届け、Engagementまでの前進を支援できること。
  - Web / Email / In-product / Intercom 等の複数接点で、顧客理解に一貫性のある提案を行い、過剰でも遅延でもない介入を実現できること。
  - 施策（Sent/配信）を「Action」として記録し、顧客変化（Signal/Usage/Phase）との接続を通じて学習・改善できること。
  - CSM担当顧客に対しても、CSMの攻略方針と矛盾しない補完介入（例：オンボーディング促進、活用コンテンツ供給、停滞時のリマインド）を連動して行えること。

### 4.3 Tertiary User（v1）
- GTM / Enablement（製品価値の翻訳・利用最大化）
  - 開発部門が生み出す製品能力（機能・アップデート）を、顧客の文脈に沿った価値として翻訳し、正しい理解と利用を促進すること。
  - 新機能や活用方法を「誰に・どのタイミングで・どう伝えるべきか」を設計し、顧客の期待・誤解・不安と製品の意図のギャップを埋めること。
  - 既存ユーザーには定着と高度活用を、これから利用するユーザーには導入理解を促し、結果として組織内での利用価値の最大化に寄与すること。
  - CSM/CRM/Supportが得た顧客のSignal・ログの学びを、Enablement（教材/テンプレ/導線/コミュニケーション）に反映し、再現可能な価値提供を作ること。

## 5. 解決すべき機会（Opportunity）
本CXMの価値は「顧客理解（Log/Signal/Phase）→適切な介入→学習」のループを回すことにある。
このとき介入は、単なるメッセージ送信に留まらず、顧客の状態に合わせた **コンテンツ（記事/資料/ガイド/手順/ヘルプ導線）** をタイムリーに届けることで初めて前進を生む。
そこでv1では、AIを“判断の補助”だけでなく **コンテンツ作成と提示のエンジン** として活用し、各ロールの介入品質とスピードを同時に引き上げる。

### 5.1 顧客理解を「コンテンツに変換する機会」
- ログ（Slack/Intercom/議事録/チケット/利用ログ）から抽出されたSignalを、単なる監視指標で終わらせず、
  「この顧客が次に必要としている説明・資料・手順」を特定し、コンテンツとして提示できる。
- 顧客の詰まり（Confusion/Tech trouble/未解決チケット）を検知した時点で、
  その状況に最適化された回答テンプレ、手順書、FAQ候補、案内導線をAIで生成し、即座に介入に接続できる。
- 顧客の機会（Opportunity：拡張兆候/成果言及/利用増）を検知した時点で、
  事例資料、上長報告用サマリ、他部署展開用の説明スライド草案などをAIで生成し、CSMの組織攻略に接続できる。
※v1ではSignal TaxonomyはHealth/Risk/Opportunityを主軸とし、Confusion等の体験ラベルはログ/チケット分類として扱い、AI拡張でSignal化する。


### 5.2 CSMの介入を「資料生成込み」で高速化する機会
CSMは導入相談〜オンボーディング〜活用〜Expandまで伴走するため、毎回ゼロから資料や説明を作るとスケールしない。
CXM上で顧客理解が取れていることを前提に、AIにより以下を“半自動生成”できる余地がある。
- 顧客状況サマリー（1分/3分/詳細）と次アクション案
- 導入計画（オンボーディングロードマップ）・チェックリスト・ToDo
- 顧客の利用状況に合わせた「次に見るべき機能/設定/活用ステップ」ガイド
- 成果共有用の報告資料（社内共有/上長向け/経営向けの粒度違い）
- Expand用の提案骨子（他部署展開の価値仮説・想定ユースケース・体制案）

### 5.3 Supportの解決を「即時ナレッジ化」する機会
Supportは瞬間的な解決体験を提供し、顧客中心文化を体現する役割を担う。
AI活用により、解決スピードと再発防止を同時に改善できる。
- チケット/問い合わせ文脈を自動要約し、最短で状況把握できる
- 既存FAQ/過去事例から該当候補を提示し、回答案を自動生成できる
- 回答をそのまま「FAQ記事ドラフト」「ヘルプセンター記事ドラフト」「マクロ返信テンプレ」に変換できる
- 問い合わせが多い領域をSignalとして可視化し、「ヘルプ導線の改善案」までAIで提案できる

### 5.4 CRMの介入を「動的セグメント×動的コンテンツ」で成立させる機会
CRMはCSMが見切れない顧客群・局面を補完し、規模的顧客の育成も担う。
AIにより「誰に送るか（動的セグメント）」だけでなく「何を送るか（動的コンテンツ）」まで統合できる。
- Signal/Usageに基づくセグメントごとに、最適な記事・手順・事例・短いTipsをAIで生成/再編集できる
- Web/メール/プロダクト内/Intercom等の接点ごとに、同一理解から一貫した表現へリライトできる
- 顧客の反応（open/click/reply）と利用変化を見て、次に出すコンテンツを改善できる（学習ループ）

### 5.5 コンテンツを“資産化”して再利用する機会（Playbook化）
AI生成コンテンツは単発利用ではなく、学習により資産化することで効果が最大化される。
- 生成された記事/資料/テンプレを「Playbook（成功パターン）」として保存し再利用できる
- どのSignal/Phaseに効いたコンテンツかを記録し、次回同様の顧客に適用できる
- CSM/Support/CRMの介入が“属人化せず再現可能”になる

### 5.6 v1でまず狙うべき機会（優先度）
（※ここでの優先度は開発順ではなく、CXMとして価値が出る順＝効果の大きさを示す）
1. Signal発火→推奨介入（Primary Layer）→「送るべきコンテンツ案」の生成（AI）
2. CSMのオンボーディング/成果共有/Expand資料の半自動生成（AI）
3. Supportの回答案生成→FAQ/ヘルプ記事ドラフト化（AI）
4. CRMの動的セグメントに対するコンテンツ生成・配信・改善（AI）
5. 生成コンテンツのPlaybook化（再利用と学習の基盤）

## 6. 要件（MVP / v1）
本章では、CXM v1として最低限成立させるべき要件を定義する。
なお本v1は、組織の主リソースであるCSMが担当顧客に対して価値を出せることを最優先とし、
Support/CRMが補完介入として同じ顧客前進ループに接続できることを狙う。
AIは「判断の代行」ではなく、顧客理解に基づくコンテンツ生成・編集・導線設計を加速するために用いる。
※本章は“開発順”の記述ではなく、CXMとして成立させるための要件を網羅的に定義する。

### 6.1 必須（MVP）
#### A. 顧客理解（Evidence）を壊さない
- Scope（Company / Project / User）を明示し、表示・集計・介入単位がブレないこと
- Phase（1〜10）はSSOTを前提とし、UIで「確定/推定」を明確に区別できること（推定は提案に留める）
- Timelineに統合ログ（Slack/Intercom/Chatwork/議事録/チケット/利用ログ）を表示し、
  各ログにSignalタグ・重要度・関連Actionリンクを付与できること

#### B. Signal監視（AIなし版）によるRisk/Opportunity/Healthの検知
- AIなしで成立するSignal Taxonomy（Health/Risk/Opportunity）をMVPセット（例：12〜15個）で運用可能にする
- Signal発火をAnalysis（俯瞰）とCompany Detail（司令塔）で確認できること
- Signalごとに推奨介入レイヤー（CSM/Support/CRM/GTMのPrimary）を紐づけられること

#### C. Action（介入）を一貫した形で記録できる（SentのCXM統合）
- Sent（配信）はMessage ActionとしてAction体系に統合する（送信ログの独立管理にしない）
- Actionには最低限、Owner Role / Scope / Target / Evidence を保持する
  - Owner Role: CSM / Support / CRM / GTM
  - Scope: Company / Project / User
  - Target: Risk↓ / Opportunity↑ / Health↑ / Phase→（いずれか）
  - Evidence: Signal ID / Logリンク（複数可）

#### D. AIによるコンテンツ生成（介入品質と速度の改善）
- Signal/ログ/Phaseの文脈を入力に、以下の「介入用コンテンツ案」をAIが生成できること（ドラフトで良い）
  - CSM向け：顧客サマリー（1分/3分/詳細）、オンボーディング手順、成果共有文、報告資料骨子、Expand提案骨子
  - Support向け：回答案、再現手順、FAQドラフト、ヘルプ導線案内文
  - CRM向け：セグメント別メール/インプロダクト/Intercom文面、記事要約、Tips、CTA文言
- 生成物は「承認/編集」してから利用できる（自動送信はしない）

#### E. 最低限の効果観測（Impact）
- Action（特にSent）に対して、反応指標（open/click/reply等）を記録できる
- 可能であれば、配信後の利用変化（Usageの変化）を同一画面で確認できる（7d/14dのどちらか）

### 6.2 v1（MVPの次に狙う拡張）
- AI提案カードの標準化（根拠→Primary Layer→Action→Target→期待期間）
- Playbook（コンテンツ資産化）機能：生成コンテンツをテンプレとして保存し再利用できる
- Web接触/プロダクト内接触の統合（接点の一貫性を上げる）
- Supportのナレッジ更新フロー（FAQ/ガイドの更新）をCXM上で完結できる
- Stakeholder mini-mapの運用強化（更新ルール、接触ログの自動集計）

### 6.3 非目標（明確化）
- Phase（SSOT）の自動上書き
- AIによる意思決定の自動化（AIは提案まで、人が確定）
- すべてのチャネル・すべてのデータソースの完全統合（v1では段階的に）

## 7. 評価指標（KPI / Leading）
本CXMは「顧客の前進を加速する」ための基盤であるため、
評価は“送った数”ではなく「顧客変化に接続できたか」で定義する。
v1では特に、CSMが価値を感じ、Support/CRMが補完介入として機能し始めることを重視する。

### 7.1 Leading（先行指標：運用が回っているか）
- CSMの週次利用率（Company Detail閲覧/意思決定/Action作成の回数）
- Signal確認→Action作成までのリードタイム（中央値）
- AI生成コンテンツの採用率（提案→承認→使用）
- Timelineの証拠リンク率（ActionにEvidenceが紐づいている割合）
- Supportの初動時間（1st response）と、CXM内で文脈参照した割合

### 7.2 Outcome（結果指標：顧客が前進したか）
- Risk Signalの改善率（例：Usage_dropの回復、未解決チケット減）
- Opportunity Signalの顕在化率（例：利用拡張兆候、成果言及の増）
- Usageの改善（L7 ActiveDays増、主要機能利用の広がり）
- Phase前進（SSOT更新）または前進兆候（Signal/Usage）の発生数
- Support起因の再発率低下（同種問い合わせの減少/再発間隔）

### 7.3 品質指標（コンテンツが“刺さったか”）
- 配信反応率（open/click/reply）
- 反応後の行動変化率（クリック後の利用増、ヘルプ閲覧後のチケット減）
- AI生成コンテンツの編集率（編集された上で採用されるのが健全、編集ゼロが正解ではない）
