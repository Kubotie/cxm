# Source Context別 Compose初期表示仕様

## UI設計思想

> **Compose は同じ画面でも、source context ごとに最初の顔を変える**

- **Inbox由来** → 返信文脈
- **Company / Project / User 由来** → 対応文脈
- **Audience由来** → 一括施策文脈
- **Actions / Content / Library 由来** → 業務成果物を送信用に変換した文脈

**重要原則:**
- source contextが変われば、最初に見るべき文脈も変わる
- どこから来たかで、トーン・文面・優先表示項目が異なる
- 文脈が切れないように、source固有の情報を常に上部に表示する

---

## 初期CTA文言ルール

### ✅ AIが作成した初期文面の場合
- **「承認」** - AI案を承認して次のステップへ
- **「差し戻し」** - AI案を却下してDraftに戻す
- **「レビュー依頼」** - 他の人にレビューを依頼

### ✅ 人が編集した後の場合
- **「確認」** - 内容を確認して次へ
- **「確認依頼」** - 他の人に確認を依頼
- **「内容を確認」** - 詳細を確認
- **「送信前確認」** - 最終確認
- **「修正依頼」** - 修正を依頼

**ルール:**
- AI案には「承認 / 差し戻し」
- 人手編集後は「確認系」に切り替える

## 1. From Inbox

### 📋 Source Context
問い合わせ・アラート・チケットへの返信対応

---

### 初期表示

#### **上部文脈カード（最優先表示）**
```
┌─────────────────────────────────────────────┐
│ 📧 From Inbox                              │
│ ─────────────────────────────────────────── │
│ 問い合わせへの返信                           │
│                                             │
│ 【件名】ログインできない                     │
│ 【受信】2026-03-16 14:30 (2時間前)          │
│ 【チャネル】Email from support@example.com  │
│ 【緊急度】🔴 High                           │
│ ─────────────────────────────────────────── │
│ 【関連】                                    │
│ Company: テクノロジーイノベーション          │
│ Project: プロジェクトA                      │
│ User: 田中太郎 (admin)                      │
└─────────────────────────────────────────────┘
```

#### **優先表示項目**
1. ✅ **元問い合わせ要約**（折りたたみ可能、全文表示）
   - タイトル / 件名
   - 本文全文
   - 受信日時
   - 経過時間

2. ✅ **Urgency（緊急度）**
   - 視覚的なバッジ（🔴 High / 🟡 Medium / 🟢 Low）
   - 緊急度の理由

3. ✅ **返信対象**
   - 返信先ユーザー名・Email
   - 所属Company / Project
   - Same channel / Alternative channel

4. ✅ **返信タイプ**
   - 返信 / 情報案内 / 技術サポート / エスカレーション
   - 対応方針の明示

5. ✅ **関連Evidence**
   - 過去のやりとり履歴
   - 関連FAQ / Help
   - 技術ドキュメント

---

### 初期文面

#### **件名（Subject）**
```
Re: ログインできない
```

#### **本文（Body）**
```
田中様

お問い合わせいただきありがとうございます。

ログインできない件について確認させていただきました。

【原因】
お使いのアカウントで一時的なロックがかかっている可能性があります。

【解決方法】
以下の手順でパスワードリセットをお試しください：
1. ログイン画面の「パスワードを忘れた方」をクリック
2. 登録されているEmailアドレスを入力
3. 受信したメールのリンクから新しいパスワードを設定

【参考】
ログイントラブルシューティング: https://help.example.com/login

上記手順でも解決しない場合は、お手数ですが再度ご連絡ください。

よろしくお願いいたします。

カスタマーサクセスチーム
株式会社Example
```

#### **文面の特徴**
- ✅ **回答的・説明的なトーン**
- ✅ **元問い合わせに対する直接的な回答**
- ✅ **具体的な解決手順**
- ✅ **FAQ / Help リンク**
- ✅ **署名付き（個別対応感）**

---

### 初期CTA

#### **AI生成直後の場合**
```
┌──────────────────────────────────────┐
│ [承認]  [差し戻し]  [レビュー依頼]   │
└──────────────────────────────────────┘
```

#### **人手編集後の場合**
```
┌──────────────────────────────────────┐
│ [内容を確認]  [修正依頼]  [送信]     │
└──────────────────────────────────────┘
```

#### **常時表示CTA**
- 📄 **「元の問い合わせを見る」** - Inbox内の該当itemへ
- 🔙 **「Inboxに戻る」** - Inbox一覧へ
- 📝 **「下書き保存」** - Draft状態で保存

---

### 優先表示項目（レイアウト）

#### **左サイドバー上部**
- 元問い合わせ全文
- 受信日時 / 経過時間
- Urgency
- Source channel

#### **左サイドバー中部**
- 返信対象ユーザー
- 返信方法（Same / Alternative channel）

#### **中央エディタ上部**
- 返信タイプバッジ（返信 / 案内 / サポート）
- 推奨返信意図

#### **右サイドバー**
- 関連Evidence
- 過去のやりとり履歴
- FAQ / Help リンク

#### **左サイドバー下部**
- Unresolved recipient（あれば）
- 送信準備状況

---

### 避けるべき見せ方

❌ **マーケティング施策っぽい見た目**
- 一括送信感を出さない
- 「キャンペーン」「セグメント」などの用語を使わない

❌ **会社全体施策に見える表示**
- Company単位の論点を前面に出さない
- Project横断分析を表示しない

❌ **テンプレート感**
- 「このテンプレートを使用」などの表示を避ける
- 差し込み変数を見せすぎない

✅ **理想的な見せ方**
- 「この問い合わせに対する返信」であることが一目で分かる
- 個別対応感が強い
- 返信の妥当性を確認しやすい

- Body（変数含む、編集可能）
- 変数の強調表示

#### **中央エディタ下部**
- Sample Output
  - 変数埋め込み後のプレビュー
  - 複数ケースの切り替え

#### **右サイドバー**
- Template情報
  - ベストプラクティス
  - 推奨ユースケース
  - 過去の使用実績

### 避けるべき見せ方

❌ **完成されたContentに見える表示**
- 「すでに完成している」印象を与えない
- Contentとの違いを明確にする

❌ **問い合わせ返信に見える表示**
- 「Re:」「返信」などの表現を避ける

❌ **変数が分かりにくい表示**
- 必須変数 / 任意変数の区別を明確に
- 変数入力状況が一目で分かるようにする

✅ **理想的な見せ方**
- 再利用資産（テンプレート）を使っていることが明確
- 変数設定が必要であることが分かる
- プレビューで実際の出力が確認できる
- Libraryから実送信用に変換している過程が見える

## まとめ: Source Context別の優先順位

### 🔴 最優先表示（どのSource Contextでも）
1. **Source Context表示**（上部文脈カード）
2. **送信意図・目的**
3. **Linked Company / Project / User**
4. **初期CTA**（AI生成 / 人手編集で切り替え）

### 🟡 Source Context固有の優先表示

| Source Context | 最優先表示項目 |
|---------------|---------------|
| **Inbox** | 元問い合わせ全文、Urgency、返信タイプ |
| **Company** | Company Phase / Health、Project横断傾向、主要関係者 |
| **Project** | Project A-Phase / Insight、Linked Users、Risk/Opportunity |
| **User** | Recent Activity、Support History、個別支援文脈 |
| **Audience** | ⚠️対象数、Audience条件、Resolved/Unresolved内訳 |
| **Actions** | Action Purpose / Owner / Due、変換意図 |
| **Content** | Variables、Sample Output、プレビュー |
| **Library** | Template説明、Variables設定、Applicable Scope |

### 🟢 文面トーンの違い

| Source Context | 文面トーン | 署名 |
|---------------|-----------|------|
| **Inbox** | 回答的・説明的 | 個人名付き |
| **Company** | フォーマル・全社的 | 個人名付き |
| **Project** | やや説明的・支援的 | 個人名付き |
| **User** | カジュアル～やや丁寧・個別的 | 個人名付き |
| **Audience** | やや説明的・一括感 | チーム名 / 個人名 |
| **Actions** | やや丁寧・依頼的 | 個人名付き |
| **Content** | フォーマル・定型的 | チーム名 |
| **Library** | 変数埋め込み前・構造的 | チーム名 / 個人名 |

## 実装時の注意点

### ✅ URLパラメータ設計
```typescript
// Compose画面へのURL例
/outbound/{outboundId}/compose?sourceContext=inbox&sourceId={inboxItemId}
/outbound/{outboundId}/compose?sourceContext=company&sourceId={companyId}
/outbound/{outboundId}/compose?sourceContext=audience&sourceId={clusterId}
```

### ✅ 初期状態の判定ロジック
```typescript
// AI生成直後 → 「承認」「差し戻し」「レビュー依頼」
// 人手編集後 → 「確認」「確認依頼」「修正する」「送信」

if (outbound.lastEditedBy === 'AI' && outbound.editCount === 0) {
  showCTAs(['承認', '差し戻し', 'レビュー依頼']);
} else {
  showCTAs(['内容を確認', '確認依頼', '送信']);
}
```

### ✅ Source Context別の文脈カード表示
```typescript
switch (sourceContext) {
  case 'inbox':
    return <InboxContextCard inquiry={inquiry} />;
  case 'company':
    return <CompanyContextCard company={company} />;
  case 'audience':
    return <AudienceContextCard cluster={cluster} targetCount={45} />;
  // ... その他
}
```

これで、**Composeが同じ画面でも、Source Contextごとに最初の顔を変える設計**が完成しました！
